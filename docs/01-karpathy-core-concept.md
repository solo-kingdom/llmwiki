# Karpathy LLM Wiki 核心概念

> 来源：[karpathy/llm-wiki.md](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

## 与传统 RAG 的本质区别

```
传统 RAG: 查询时 → 检索碎片 → 拼凑答案 → 下次重新开始
LLM Wiki: 摄取时 → 编译知识 → 写入 Wiki → 知识常驻累积
```

核心差异：**Wiki 是一个持久化的、可累积的知识产物**。交叉引用已经有、矛盾已经被标注、综合已经反映了所有已读内容。Wiki 随每个新增源和每次查询变得越来越丰富。

## 三层架构

```
┌──────────────────┐
│  Raw Sources     │  源文件（PDF、文章、笔记、图片）
│  (不可变)         │  LLM 只读，永不修改，真理源
└────────┬─────────┘
         │  LLM 摄取
         ▼
┌──────────────────┐
│  The Wiki        │  LLM 生成的 Markdown 文件
│  (LLM 拥有)       │  overview / entities / concepts /
│                  │  synthesis / comparisons / log
└────────┬─────────┘
         │  约定
         ▼
┌──────────────────┐
│  The Schema      │  告诉 LLM 如何组织和维护 Wiki 的规则文档
│  (人与 LLM 共演化) │  如 CLAUDE.md / AGENTS.md / schema.md
└──────────────────┘
```

## 三大操作

### 1. Ingest（摄取）
- 将新源文件放入 raw 目录
- LLM 读取 → 写摘要页 → 更新 index → 更新相关实体/概念页 → 追加 log
- **单个源可能触及 10-15 个 wiki 页面**
- 人可以参与引导，也可批量处理

### 2. Query（查询）
- 针对 Wiki 提问
- LLM 搜索相关页面 → 阅读 → 综合回答，带引用
- 输出可以是 Markdown 页、对比表、幻灯片（Marp）、图表（matplotlib）
- **好的回答可以归档回 Wiki 作为新页面**（不会消失在聊天记录中）

### 3. Lint（维护）
- 周期性健康检查：
  - 页面间的矛盾
  - 被新源替代的过时声明
  - 孤立页面（无入链）
  - 被提及但缺少独立页面的重要概念
  - 缺失的交叉引用
  - 可通过网络搜索填补的数据空白

## 两个导航文件

### index.md（内容目录）
- 按类别组织（实体、概念、源文件等）
- 每个条目包含链接、一行摘要、可选元数据（日期、源计数）
- LLM 每次摄取后更新
- 查询时 LLM 先读 index 找相关页面，再深入阅读
- 在中等规模（~100 源，~数百页）时效果很好，避免引入 embedding RAG 基础设施

### log.md（操作日志）
- 仅追加的时间线记录
- 每个条目以一致前缀开头（如 `## [2026-04-02] ingest | Article Title`）
- 可用 grep 解析：`grep "^## \[" log.md | tail -5`
- 帮助 LLM 理解最近做了什么

## 角色分工

| 角色 | 职责 |
|------|------|
| **人** | 筛选源、引导分析、提好问题、思考含义 |
| **LLM** | 所有其他工作：摘要、交叉引用、归档、簿记 |

人的工作是 curation（策展），LLM 的工作是 maintenance（维护）。

## 为什么这能工作

维护知识库的繁琐部分不是阅读或思考——而是簿记。更新交叉引用、保持摘要新鲜、标记新旧数据矛盾、维护数十页的一致性。人类放弃 Wiki 是因为维护负担增长快于价值增长。

LLM 不会感到无聊、不会忘记更新交叉引用、可以一次触及 15 个文件。**维护成本降至近零**。

## 理论渊源

与 Vannevar Bush 的 Memex (1945) 精神相通——一个私人的、主动策展的知识仓库，文档之间的关联和文档本身一样有价值。Bush 无法解决"谁来做维护"的问题。LLM 解决了。
