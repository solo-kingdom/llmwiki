## Context

当前前端是功能先行演进的结果：`App.tsx` 通过 Tabs 组件管理页面切换，Ingest 侧同时存在顶部模型下拉、左侧 Session 列表和底部发送区，Jobs/Settings 页面容器策略不一致，Wiki 缺少右侧大纲。问题核心不是单点样式，而是缺少统一布局骨架与统一交互优先级，导致视觉噪声和认知负担偏高。

本次变更聚焦前端结构与交互重排，不引入新的后端 API。已有 `AppContext` 能提供 provider instances、models、session 列表与切换能力，足以支撑“按钮 + 模态框”的交互形态调整。

## Goals / Non-Goals

**Goals:**
- 建立统一页面骨架：语义化主导航 + 一致化页面容器宽度与居中策略。
- 将模型选择聚合到发送主路径附近，并以模态框承载复杂选择流程。
- 将 Session 管理从左侧重型列表降维为轻量入口（切换/新增双按钮）。
- 将 Wiki 升级为三栏布局（目录树/正文/大纲），提升文档定位效率。
- 在不改动核心后端接口的前提下完成 UI 体验重构。

**Non-Goals:**
- 不新增 provider/model 管理后端能力。
- 不重写聊天消息流、归档流程或 ingest pipeline 行为。
- 不覆盖移动端全面重设计（仅保证结构不破坏现有响应式基础）。
- 不在本次变更内引入主题系统重构或全量视觉品牌升级。

## Decisions

### D1: 主导航从 Tabs 视觉切换为按钮式导航，状态管理仍保留单一 `view`

**决策**: 在 `App.tsx` 中移除 Tabs 视觉组件作为主导航承载，改为 header 内按钮组（active 态按钮 + 普通态按钮）；页面切换仍由单一 `view` 状态控制。

**理由**:
- Tabs 更适合局部内容切换，不适合全局路由语义。
- 不引入 React Router，避免超出本次 UI 重构范围。
- 低风险：复用现有视图组件装配方式即可。

**备选方案**:
- 直接引入路由系统并做 URL 状态持久化。优点是可分享链接，缺点是改动面大、与本次目标不匹配，暂不采用。

### D2: 模型选择采用“发送区附近入口 + 模态框”

**决策**: 移除 `IngestChat` 顶部 instance/model 下拉，新增发送区附近“模型”按钮；点击打开模态框，完成 instance + model 选择并调用现有 `updateSessionLLM`。

**理由**:
- 将模型选择放到“发送消息前最后一步”更符合任务流。
- 模态框可容纳二级选择、空状态和错误提示，避免顶部拥挤。
- 复用现有 API 与状态，不新增后端复杂度。

**备选方案**:
- 保留顶部下拉并仅调整样式。无法解决交互断裂问题，拒绝。

### D3: Session 管理入口轻量化

**决策**: 将左侧 Session 列表弱化为“切换按钮 + 新建按钮”入口；切换按钮打开会话列表弹层，新建按钮直接创建会话。

**理由**:
- 释放横向空间给聊天主内容。
- 与模型选择方式统一（入口按钮 + 弹层）。
- 保持 `listSessions` / `switchSession` / `createSession` 数据流不变。

**备选方案**:
- 保留可折叠左栏为默认形态。仍会造成视觉权重过高，暂不采用。

### D4: Jobs 与 Settings 使用统一内容容器 token

**决策**: 抽象统一容器样式（例如 `max-w-5xl mx-auto px-6`），同时用于 Jobs 和 Settings 页面主内容。

**理由**:
- 一次性解决“过宽 + 不居中 + 页面尺度不一致”问题。
- 可为后续页面一致性优化复用。

### D5: Wiki 采用三栏布局并新增右侧大纲能力

**决策**: 复用现有左侧 `Sidebar` 与中间 `DocumentViewer`，新增右侧 `Outline` 面板；大纲由当前文档标题结构生成并支持点击滚动。

**理由**:
- 与参考产品布局一致，显著提升长文导航效率。
- 与现有 markdown 渲染链兼容，变更集中在前端组合层。

**备选方案**:
- 在中间正文顶部显示扁平目录。导航效率不如右栏持续可见，暂不采用。

## Risks / Trade-offs

- **[交互迁移学习成本]** 用户对旧位置形成肌肉记忆 → 通过清晰图标、灰色状态标签和一致按钮语义降低迁移摩擦。
- **[状态同步复杂度上升]** 模态框选择与会话切换可能出现状态竞争 → 统一以 `AppContext` 为单一数据源，模态框仅维护临时草稿状态。
- **[Wiki 三栏压缩正文宽度]** 小屏可能可读性下降 → 桌面端三栏，窄屏按需折叠右栏或转抽屉。
- **[范围膨胀风险]** UI 优化易演变为全面重构 → 严格限制在 proposal 定义的 capability 与页面范围内。
