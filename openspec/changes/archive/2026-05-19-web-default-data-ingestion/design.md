## Context

当前系统在能力层面已具备部分摄取基础（队列、分层源处理、工作区重建原则），但入口层面仍以“文档浏览”为主：Web 仅提供 Wiki/Settings，服务端缺少面向 Web 的 ingest API，用户无法在单一界面完成“提交→处理→追踪→纠错”的完整流程。

本变更目标是将 Web 提升为默认摄取入口（Web-first ingest），并把对话式输入、文本输入、文件上传统一抽象为 ingest job。该设计需保持既有约束：文件系统真理、SQLite 衍生索引、可重建一致性、单进程架构。

## Goals / Non-Goals

**Goals:**
- 建立 Ingest Hub 作为 Web 默认主入口，并保留现有浏览能力。
- 定义统一摄取任务模型（job lifecycle + 错误语义 + 重试能力）。
- 支持三类提交方式：对话式摄入、文本提交、文件上传。
- 将格式能力与运行时依赖状态可视化（capabilities 联动）。
- 保证摄入落盘与索引更新遵循文件真理边界。

**Non-Goals:**
- 本次不引入新的外部 OCR/解压引擎实现（如 Tesseract、专用沙箱解压服务）。
- 本次不重构 MCP 工具协议，仅新增/扩展 HTTP 与 Web 的摄取工作流。
- 本次不覆盖高级协同编辑能力（多人会话、共享草稿）。

## Decisions

### 1) 入口决策：Web 默认落在 Ingest Hub
- 决策：Web 首屏默认进入 Ingest Hub；Wiki 浏览转为并列导航而非默认视图。
- 原因：与“Web 作为默认数据摄入入口”的产品目标一致，降低新用户路径成本。
- 备选：保留 Wiki 默认页 + 强提示卡片。该方案改动小，但不满足“默认入口”语义。

### 2) 模型决策：统一使用 ingest job（异步）
- 决策：三种输入统一转换为 ingest job，状态机为 `queued/running/succeeded/failed/cancelled`。
- 原因：统一可观测、重试、诊断模型，避免按输入类型分裂后端流程。
- 备选：文本同步、文件异步混合模型。该方案实现快，但状态语义不一致，前端复杂度更高。

### 3) 对话式摄入决策：先生成草稿，再确认入队
- 决策：对话输入先形成 ingest draft（可编辑），用户确认后创建 job。
- 原因：降低误摄入风险，提升可控性与可审计性。
- 备选：对话消息自动入队。该方案自动化强，但误触发成本高、纠错困难。

### 4) 文件真理边界决策：原始输入先落盘，再进入索引/处理
- 决策：文本提交与上传文件都先持久化到工作区规范位置（如 raw/sources 或约定 inbox），再进入处理流水。
- 原因：符合“filesystem source of truth”与“删库可重建”原则。
- 备选：仅保存到 DB 二进制字段。违反现有真理边界，不采纳。

### 5) 格式策略决策：能力分层 + 结构化失败
- 决策：沿用 Tier A/B/C 能力模型；不可提取时返回明确 `error_code/missing_dependency/remediation`。
- 原因：用户可诊断、可修复，且与已有 capabilities 端点契合。
- 备选：统一报“处理失败”。可用性差，无法指导运维动作。

## Risks / Trade-offs

- [风险] 默认入口迁移可能影响老用户使用习惯  
  → Mitigation：保留 Wiki 快速入口与最近访问列表，提供一次性引导。

- [风险] 上传大文件/压缩包导致单进程资源压力上升  
  → Mitigation：限制单任务大小与并发；压缩包采用逐文件解包与配额控制。

- [风险] 对话草稿与最终任务语义不清导致误解  
  → Mitigation：UI 强化“草稿未入队/已入队”状态标识，并展示确认步骤。

- [风险] 依赖缺失（pdftotext/libreoffice）导致体验落差  
  → Mitigation：提交前展示 capability 预检，失败后返回 remediation。

- [权衡] 统一异步作业会增加状态管理复杂度，但换来一致可观测性与可靠重试。

## Migration Plan

1. 新增 ingest API 与 job 存储/查询接口，先保证后端可独立验证。
2. Web 引入 Ingest Hub 与三类提交流程，默认路由切换到摄取入口。
3. 灰度保留 Wiki 入口，补齐引导与兼容导航。
4. 补充端到端验收：文本、对话草稿转任务、文件上传、失败重试、能力降级提示。
5. 回滚策略：若入口迁移导致严重问题，可将默认视图临时切回 Wiki，保留新 API 不对外暴露。

## Open Questions

- 压缩包支持边界：仅 zip，还是包含 rar/7z（及其依赖）？
- 对话草稿保存周期：会话级临时缓存，还是持久化草稿实体？
- 任务取消语义：仅取消未开始任务，还是支持运行中中断？
- 多文件上传失败策略：部分成功是否允许提交，如何回显汇总结果？
