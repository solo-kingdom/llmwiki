## Context

当前工作台默认入口已是聊天摄入（`IngestChat`），并通过会话归档进入 pipeline。历史上也存在一次性文本摄入与文件上传能力（`/api/v1/ingest/jobs/text`、`/api/v1/ingest/jobs/upload`），但缺少一个清晰的“Raw 直投”页面承载，导致用户在“探索式摄入”和“直接归档”之间切换成本高。

本次设计目标是在不破坏现有会话归档流程的前提下，明确拆分两条路径：
- Chat：需要和模型探索时使用
- Ingest：已有原始数据时直接提交

## Goals / Non-Goals

**Goals**
- 将 Chat 与 Raw Ingest 在导航层面明确分离
- Raw Ingest 支持“多文本块 + 多文件”混合输入
- 一次点击触发归档提交，并提供可追踪任务反馈
- 最大化复用现有 API 与状态管理，降低改动风险

**Non-Goals**
- 不改动 ingest pipeline 的核心处理逻辑
- 不在本次引入新的后端 batch 事务接口
- 不改变 Jobs 页的数据模型与生命周期定义

## Decisions

### 1) 路由与导航语义拆分

**决策**
- 导航项由 `Ingest / Jobs / Timeline / Logs / Settings` 调整为 `Chat / Ingest / Jobs / Timeline / Logs / Settings`
- 默认路径 `/` 映射到 Chat（兼容现有习惯）
- 新增 `/ingest` 映射到 Raw Ingest 页面

**理由**
- 保留默认入口行为，避免老用户路径失效
- 通过显式命名减少“当前页面是否需要对话”的认知负担

### 2) Raw Ingest 页面交互模型

**决策**
- 页面包含三个区域：
  - 文件区：支持多文件选择与拖放
  - 文本块区：支持动态新增/删除文本块，每块可填写标题和正文
  - 提交区：显示批次标题/来源（可选）与 `直接归档` 按钮

**理由**
- 文件与文本是最常见的两类原始输入
- 文本块化优于单一大文本框，便于用户分段组织材料来源

### 3) 提交编排策略（首版）

**决策**
- 文本块在前端拼接为一份标准 markdown（保留块标题与分隔）后，调用一次 `createTextIngestJob`
- 文件调用 `uploadIngestJobs` 批量上传
- 两类提交可同次触发，最终汇总 accepted/rejected 与 job id 列表反馈给用户

**理由**
- 不增加后端 API 成本即可实现“统一提交体验”
- 通过“一个文本 job + 多文件 job”维持可观测性与失败隔离

### 4) 提交约束与反馈

**决策**
- 至少存在一个非空文本块或一个文件才能点击 `直接归档`
- 提交后显示摘要反馈（成功数、失败数、失败原因），并提供 `查看 Jobs` 导航动作

**理由**
- 防止空提交
- 将执行反馈统一收敛到 Jobs 页，符合现有任务中心设计

### 5) 与 Chat 页职责边界

**决策**
- Chat 页继续负责“会话探索 + 归档”
- Raw Ingest 页不提供聊天消息流与会话切换能力，仅承担原始数据直投

**理由**
- 形成明确职责边界，避免新页面再次演化成“第二个 Chat”

## Risks / Trade-offs

- **多文本块拼接质量**：拼接策略过弱可能影响下游理解；通过统一 markdown 结构与块级标题缓解
- **混合提交的原子性**：文本与文件分别提交，失败时可能部分成功；通过清晰反馈与 Jobs 可重试机制兜底
- **导航迁移成本**：文案从 Ingest 改为 Chat 会影响已有测试与文档；需一次性同步更新
