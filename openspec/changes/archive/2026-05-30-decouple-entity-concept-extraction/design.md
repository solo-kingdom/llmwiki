## Context

LLM Wiki 当前已通过 typed directories 区分 `entities/`、`concepts/`、`sources/` 等页面类型，但语义边界主要停留在“目录和 frontmatter type 是否一致”。摄入提示词要求识别实体、概念和关系，却没有要求模型在命名和页面规划时避免把具体实体与抽象概念合并为单个页面。

实际结果中出现了 `wiki/concepts/AppLovin组织裁剪方法论.md` 这样的页面：目录和 type 可能是正确的，但标题语义把具体公司 `AppLovin` 和可复用概念 `组织裁剪方法论` 绑定在一起。更合理的结构是保留 `wiki/entities/AppLovin.md`，创建或更新中性概念页 `wiki/concepts/组织裁剪方法论.md`，并通过正文中的 wikilink 描述案例关系。

本变更需要同时影响三层：

- `skills/`：作为提示词设计蓝本，沉淀实体/概念判定规则和反模式。
- `internal/ingest/prompts.go`：作为运行时系统提示词，约束分析、生成和整理阶段。
- `internal/engine` lint：作为机械检查，发现已有 wiki 中的疑似结构问题。

## Goals / Non-Goals

**Goals:**

- 让摄入分析先区分实体、概念、论点和关系，并标出疑似“实体 + 概念”混绑项。
- 让生成阶段优先创建中性概念页，通过 wikilink 连接实体页，而不是创建实体绑定型概念标题。
- 让 organize/lint 能报告已有 `concepts/` 页面中疑似以实体名开头并拼接抽象术语的页面。
- 同步更新中文和英文 skill 蓝本，以及中文和英文运行时提示词。
- 为新增 lint 行为和提示词约束补充测试，防止后续回退。

**Non-Goals:**

- 不自动移动、重命名或删除用户已有页面；lint 只报告问题和建议。
- 不引入外部 NLP/NER 依赖；实体识别基于已有 wiki entity 页面和简单标题启发式。
- 不要求所有概念标题完全禁止实体名；公认专有名词或源中明确命名的术语可通过提示词和用户规则保留。
- 不实现跨文档大规模知识重构；本变更提供规则和检测，具体整理仍由后续 apply 或用户确认执行。

## Decisions

### 1. 在提示词中建立实体/概念/关系三分法

运行时分析提示词应明确：

- entity：可唯一指代现实对象，如人、组织、产品、项目。
- concept：可跨对象复用的术语、方法、框架、机制、模式或理论。
- relation：实体与概念之间的采用、提出、体现、案例、对比等关系。

分析阶段要求模型先列出候选实体和候选概念，再规划页面。对于 `AppLovin 组织裁剪方法论` 这类短语，应输出为：

- entity: `AppLovin`
- concept: `组织裁剪方法论`
- relation: `AppLovin` 是该方法论的案例或实践者

替代方案是只在生成阶段修正标题。该方案更轻量，但分析结果本身会继续把混绑短语传递给生成阶段，容易被后续步骤放大，因此不采用。

### 2. 在生成提示词中增加标题与路径自检

生成阶段在写 FILE 块前应执行一个语义自检：

- 新建 `wiki/concepts/` 页面时，标题和文件名默认不应以已有实体名作为前缀。
- 如果候选概念标题形如“实体名 + 方法/模型/文化/框架/策略/机制/理论/实践”等，应拆为中性概念页，并在正文加入实体案例链接。
- 如果该词是明确专有名词或源材料把整体当作固定术语，应在正文说明命名依据，并仍优先考虑实体页与概念页之间的链接。

替代方案是仅依赖 workspace `rules.md` 或 `.llmwiki/prompts.yaml`。这不适合作为默认质量规则，因为问题来自通用摄入行为，应该进入默认提示词。

### 3. 在 lint 中新增启发式检查，而不是修改路径分类核心

新增 lint code 建议为 `entity_concept_coupling`，severity 为 `warning`。它只在 `wiki/concepts/` 下运行，条件为：

- 当前概念页标题或文件名 stem 以某个已存在实体页标题或文件名 stem 开头。
- 剩余部分包含抽象概念后缀或关键词，如 `方法`、`方法论`、`模型`、`文化`、`框架`、`策略`、`机制`、`理论`、`范式`、`实践`、`流程`、`体系`。
- 命中的实体名长度足够具体，避免把过短词误判为实体前缀。

lint 只报告并建议：将页面拆分或重命名为中性概念，并从概念页链接实体案例、从实体页链接相关概念。

替代方案是把该规则纳入 `ValidateFrontmatter`。这会把语义质量和 frontmatter 校验混在一起，不利于维护，因此不采用。更合适的位置是 `LintWorkspace` 中基于全量页面集合的结构检查，因为它需要知道已有实体页列表。

### 4. 保持诊断输出可被 organize 模式消费

`search(mode="lint")` 和 `audit(focus="structure")` 已经会展示 lint issues。新增检查应沿用相同结构，不需要新增 API。组织模式提示词应说明此类 warning 的处理方式：优先重命名/拆分/补链，不要直接删除原页面。

## Risks / Trade-offs

- [误报专有名词] 某些概念本身可能包含公司名或人名 → 将检查设为 warning，并在提示词中允许源材料明确支持的专有术语保留。
- [实体列表不完整] 如果实体页尚未创建，lint 可能无法发现混绑 → 摄入提示词先要求实体抽取，并鼓励优先创建/更新实体页，可降低漏报。
- [中文/英文命名差异] 文件名可能用下划线、空格或大小写变体 → lint 应对标题和 stem 做基础归一化，包括去空格、下划线、连字符和大小写归一。
- [模型仍可能生成混绑标题] 提示词不是强约束 → 通过 lint 检查和测试覆盖，把问题暴露到后续整理流程。
- [已有页面迁移成本] 老页面可能已有入链 → 本变更不自动迁移；后续实施整理时需要更新链接，避免死链。

## Migration Plan

1. 更新 `skills/llmwiki-ingest` 中英文蓝本，加入实体/概念/关系判定、标题反模式和示例。
2. 更新 `internal/ingest/prompts.go` 中分析、生成和 organize 提示词。
3. 在 `internal/engine/lint.go` 增加 `entity_concept_coupling` warning 检查和测试。
4. 确保 MCP lint/audit 输出自动包含新 issue；必要时更新 lint skill 文档。
5. 手动或自动验证示例：已有 `AppLovin` 实体页时，`wiki/concepts/AppLovin组织裁剪方法论.md` 应被报告为 warning。

Rollback 策略：提示词改动可回退对应文本；lint 检查可通过移除新 code 和测试回退，不涉及数据迁移。
