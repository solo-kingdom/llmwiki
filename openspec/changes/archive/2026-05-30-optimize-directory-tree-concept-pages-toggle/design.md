## Context

当前 Wiki 左侧导航同时呈现实体列表与目录树，且目录树中的 Pages 又包含实体文档，导致信息重复与认知冲突。用户在“按知识类型理解内容”和“按页面结构浏览文档”之间缺少明确模式切换，尤其在实体数量增长后会降低定位效率。

该变更聚焦前端 Reader 侧边栏交互，不涉及后端索引结构调整。已有页面类型、文档路径和数据接口可复用，重点在于前端对同一数据集的两种视图编排与默认策略。

## Goals / Non-Goals

**Goals:**
- 在左侧导航提供清晰的“概念 / Pages”视图切换，并默认进入概念视图。
- 概念视图聚合 `entity`、`concept` 与 `overview` 内容，作为知识导览入口。
- Pages 视图继续承担文档树浏览职责，并保持与页面类型体系一致。
- 切换视图时不打断当前阅读文档与滚动上下文。

**Non-Goals:**
- 不调整 wiki 文件组织规范与后端 page type 判定规则。
- 不引入新的页面类型或重命名既有类型。
- 不改造知识图谱页面与右侧大纲面板逻辑。

## Decisions

### 决策 1：引入显式侧边栏模式切换（概念 / Pages）
- 方案：在 Reader 左侧导航顶部增加双态切换控件，状态保存在页面级 UI state。
- 原因：以最低实现复杂度消除“实体列表 + Pages 重复展示”的结构歧义。
- 备选方案：
  - 自动根据筛选条件推断模式：学习成本高，状态不可预期。
  - 保留并列双区块仅微调文案：无法根除重复信息问题。

### 决策 2：默认模式设为概念视图
- 方案：首次进入 `/wiki` 默认展示概念视图；用户可手动切换到 Pages 视图。
- 原因：更贴合“先理解知识结构、再深入文档细节”的阅读路径。
- 备选方案：
  - 默认 Pages：延续旧习惯但无法体现本次抽象层级优化目标。

### 决策 3：概念视图与 Pages 视图采用不同的数据投影
- 方案：两种视图共享同一 wiki 文档数据源，但以不同过滤与分组策略渲染：
  - 概念视图：展示 `entity`、`concept`、`overview` 集合。
  - Pages 视图：展示完整 wiki 文档树，并沿用 page type 语义与筛选行为。
- 原因：避免新增后端接口，降低实现和回归成本。
- 备选方案：
  - 新增后端聚合接口：收益有限，增加 API 与测试面。

## Risks / Trade-offs

- [风险] 视图切换后用户可能误解“筛选器作用范围”  
  → Mitigation：在模式标签与空状态文案中明确“当前为概念视图/Pages 视图”。

- [风险] 旧测试依赖“实体列表始终可见”假设而失败  
  → Mitigation：按模式拆分测试断言，补充默认模式与切换行为测试。

- [权衡] 保持数据源复用会增加前端条件渲染分支  
  → Mitigation：抽离模式判定与列表构建函数，减少组件内嵌套逻辑。

## Migration Plan

1. 更新 OpenSpec：新增侧边栏模式能力并修改 `wiki-reader-ui` 相关要求。
2. 实现前端模式状态、切换控件与两套渲染策略。
3. 调整现有单测，新增默认模式与切换场景测试。
4. 手工验收：验证默认概念视图、切换后列表内容、文档阅读连续性。
5. 回滚策略：若出现高风险回归，可临时恢复单一旧侧边栏布局（保留数据层改动不落地）。

## Open Questions

- 概念视图中的 `overview` 是否固定置顶，还是按标题字母排序参与列表。
- 是否需要记住用户上次模式（会话内/本地持久化），当前提案默认不持久化。
