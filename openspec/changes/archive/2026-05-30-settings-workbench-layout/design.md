## Context

当前 Web UI 已经形成两套明确的产品空间：管理工作台负责摄入、任务、日志、帮助和设置；Wiki reader 负责阅读、检索、图谱和文档导航。工作台通过 `WorkbenchContentShell` 使用居中的 `max-w-5xl` 内容列，Wiki reader 通过全屏 header、左侧导航、中央阅读卡片和右侧大纲实现阅读优先布局。

Settings 页面位于管理工作台内，但它的内容复杂度已经超过普通表单。页面内既有日常配置，也有 Provider 连接、规则预览、MCP JSON、后台任务参数、日志上限、处理参数和版本控制状态。继续线性堆叠会让用户难以理解优先级，也会让保存操作和高级项显得混杂。

## Goals / Non-Goals

**Goals:**

- 让 Settings 页面按用户心智组织：先处理基础和常用设置，再处理连接、自动化和高级配置。
- 保留管理工作台的 `max-w-5xl` 内容列，使 Settings 与 Jobs、Logs、Help 等管理页面保持一致。
- 明确 Wiki reader 继续采用全屏阅读布局，不为了“和工作台一样宽”而牺牲三栏阅读体验。
- 为长设置页提供清晰的保存状态、保存入口和反馈。
- 统一中文 UI 文案，减少英文裸字段造成的产品完成度问题。

**Non-Goals:**

- 不改变任何设置 API、Provider API、MCP API 或后端持久化格式。
- 不重设计 Wiki reader 的三栏阅读结构、搜索、图谱或大纲交互。
- 不引入新的 UI 组件库或布局依赖。
- 不改变工作台导航项集合。

## Decisions

### Decision 1: 工作台与 Wiki reader 保持不同宽度策略

管理工作台继续使用 `max-w-5xl` 居中内容列；Wiki reader 继续使用全屏三栏布局。两者共享 header 的卡片质感、圆角、边框、半透明背景和阴影，但不共享等宽容器。

理由：工作台页面多为管理表单、列表和对话界面，固定最大宽度能提升扫描效率；Wiki reader 的核心价值是阅读和导航，左侧目录、中央文档、右侧大纲需要利用横向空间。如果强行等宽，会压缩阅读面板或迫使侧栏隐藏，反而降低 Wiki 体验。

备选方案：将工作台扩展到全屏。缺点是设置页和日志页在大屏上会变得过宽，短字段和卡片留白过大。将 Wiki reader 限制到 `max-w-5xl`。缺点是三栏布局空间不足。

### Decision 2: Settings 页面采用分组设置架构

Settings 页面按配置意图分区，而不是按当前代码顺序平铺。建议分组为：

- 基础设置：语言和常规偏好。
- 模型与连接：Provider 实例、后台任务模型。
- 工作区规则与 MCP：规则预览、补充说明、MCP 服务器。
- 自动化与容量：Tool Loop、日志保留、处理参数。
- 版本控制：版本控制状态和历史入口。

理由：用户首先需要知道“我来这里要改什么”，分组能降低认知负担，也为未来增加设置项留下稳定位置。

备选方案：只增加卡片标题和描述，不调整结构。缺点是长页面仍然难以导航。

### Decision 3: 保存体验放到页面级状态，而非每张卡片独立保存

Settings 继续通过现有设置保存接口统一提交，但 UI 提供页面级未保存状态、固定保存入口和保存反馈。Provider 添加/编辑/删除、连接检查等即时操作继续保持局部操作，不和全局“保存设置”混淆。

理由：后端已有 Settings 保存模型，拆成多接口会扩大实现范围；页面级状态足以解决长页面保存入口难找的问题。

备选方案：每个分组独立保存。缺点是需要拆分 API 或处理部分保存失败，增加复杂度。

### Decision 4: 高级配置默认降低视觉权重

MCP JSON、处理参数、日志上限和 Tool Loop 参数属于低频或专家配置。它们应被放入清晰标记的高级区域，可折叠或以较低优先级显示；同时保留完整可编辑能力。

理由：这些能力对高级用户重要，但不应和语言、Provider 等首要任务争夺注意力。

备选方案：隐藏高级项到单独页面。缺点是会增加导航成本，且 Settings 页面仍是最自然的配置入口。

## Risks / Trade-offs

- 设置项重组可能让熟悉旧位置的用户短期内找不到项目 → 保留清晰分组标题和描述，必要时提供锚点导航。
- Sticky 保存入口可能占用小屏高度 → 在窄屏使用紧凑样式，并确保不遮挡最后一项表单内容。
- 高级项折叠可能降低可发现性 → 使用明确的“高级设置”标题和说明，而不是完全隐藏。
- 不让 Wiki reader 和工作台等宽可能被误解为“不一致” → 在 spec 中明确一致性来自 header 风格和产品语义，而不是容器宽度。
